Chapter 1

Variables & Type Inference

Kotlin replaces Java's verbose variable declarations with two keywords: val and var. The compiler infers types from the right-hand side, so you rarely write them explicitly.

val vs var

val is equivalent to Java's final — it cannot be reassigned after initialisation. var is mutable. Prefer val by default; reach for var only when you genuinely need reassignment.

final double south = 52.1;
double north = 52.4;
north = 52.5; // ok
// south = 0;  // compile error
val south = 52.1   // inferred as Double, immutable
var north = 52.4
north = 52.5       // ok
// south = 0.0     // compile error

Type inference

The type is inferred from the initialiser. You can always write it explicitly, but you usually don't need to.

Kotlin 
val query    = ""          // String
val limit    = 5           // Int
val loading  = false       // Boolean
val pois     = emptyList<Poi>()   // List<Poi>

// Suffixes disambiguate numeric literals
private var lastBearing         = 0f   // Float  (NavigationEngine.kt)
private var recalcCooldownUntil = 0L   // Long   (MapViewModel.kt)

Compile-time constants with const val

For primitive/String constants that are known at compile time, use const val inside an object or companion object. These are inlined at call sites — more efficient than static final in Java.

// NavigationEngine.java
private static final double OFF_ROUTE_THRESHOLD_METERS = 30.0;
// NavigationEngine.kt
companion object {
    private const val OFF_ROUTE_THRESHOLD_METERS = 30.0
}
🚲
In FietsNavNu: NavigationEngine.kt uses const val OFF_ROUTE_THRESHOLD_METERS = 30.0 to decide when the cyclist has drifted off the planned route. Because it's const, the compiler replaces every reference to it with the literal 30.0 — zero runtime overhead.
💡 Exercise

You need a speed-limit variable (fixed at 30 km/h) and a current-speed variable that updates every second. Which keyword should each use, and why?

Answer

Use const val SPEED_LIMIT = 30 for the fixed limit — it never changes and is known at compile time. Use var currentSpeed = 0.0 for the live speed — it must be updated on each GPS tick. Making the limit const val (rather than var) means the compiler can catch any accidental reassignment.

Chapter 2

Functions

Kotlin functions have a concise syntax, support default parameter values, named arguments, and can be written as single expressions — eliminating most Java boilerplate.

Basic syntax

Return type comes after the parameter list with a colon. When the body is a single expression, the return keyword and braces are replaced by =.

// Block body — always needs return
public double haversineMeters(LatLng a, LatLng b) {
    double r = 6371000.0;
    // ... math ...
    return 2 * r * Math.asin(Math.sqrt(h));
}
// Expression body — no return keyword
private fun haversineMeters(a: LatLng, b: LatLng): Double {
    val r = 6371000.0
    // ... math ...
    return 2 * r * asin(sqrt(h))
}

// Short version as expression body (return type inferred)
fun Double.toRadians() = this * Math.PI / 180

Default parameter values

No more overload chains. Parameters can declare a default, and callers that are happy with the default simply omit that argument.

// Two overloads just to provide a default
public RouteResult getRoute(List<Pair<Double,Double>> waypoints) {
    return getRoute(waypoints, "bike");
}
public RouteResult getRoute(List<Pair<Double,Double>> waypoints, String profile) {
    // real implementation
}
// RouteRepository.kt — one function, profile defaults to "bike"
suspend fun getRoute(
    waypoints: List<Pair<Double, Double>>,
    profile: String = "bike"
): RouteResult { ... }

// Callers can omit profile:
repository.getRoute(listOf(52.37 to 4.89, 52.40 to 4.91))

Named arguments

Call sites can name each argument, making them self-documenting and allowing any order.

Kotlin — MapViewModel.kt 
repository.getRoute(
    fromLat = from.lat.toDouble(),
    fromLon = from.lon.toDouble(),
    toLat   = to.lat.toDouble(),
    toLon   = to.lon.toDouble(),
    profile = profile
)
Compare this to the Java call: getRoute(from.getLat(), from.getLon(), to.getLat(), to.getLon(), profile) — four bare double values in a row. Swap two and the bug is invisible at the call site.

Retrofit interfaces — suspend functions

Retrofit API interfaces with suspend functions are a perfect fit for expression bodies. The suspend keyword is covered in Chapter 9.

Kotlin — GraphHopperApi.kt & NominatimApi.kt 
interface GraphHopperApi {
    @POST("route")
    suspend fun getRoute(@Body request: GraphHopperRequest): GraphHopperResponse
}

interface NominatimApi {
    @GET("search")
    suspend fun search(
        @Query("q") query: String,
        @Query("format") format: String = "json",
        @Query("limit") limit: Int = 5,
        @Query("countrycodes") countryCodes: String = "nl"
    ): List<NominatimResult>
}

Extension functions

You can add functions to any class — including ones you don't own — without subclassing. The function receives the object as this.

Kotlin 
// Haversine distance as an extension on LatLng
fun LatLng.distanceTo(other: LatLng): Double {
    val r = 6371000.0
    val lat1 = Math.toRadians(this.latitude)
    val lat2 = Math.toRadians(other.latitude)
    val dLat = Math.toRadians(other.latitude - this.latitude)
    val dLon = Math.toRadians(other.longitude - this.longitude)
    val h = sin(dLat / 2).pow(2) + cos(lat1) * cos(lat2) * sin(dLon / 2).pow(2)
    return 2 * r * asin(sqrt(h))
}

// Reads naturally at the call site:
val dist = snappedPoint.distanceTo(route[i + 1])
🚲
In FietsNavNu: haversineMeters is duplicated as a private function in NavigationEngine.kt, PoiServiceRepository.kt, and MapViewModel.kt. Refactoring it into a single extension function on LatLng would eliminate all three copies.
💡 Exercise

RouteRepository.kt has two getRoute functions — one that takes a list of waypoints and one that takes individual lat/lon pairs. How does the second call the first? Could you express the two as a single function with defaults?

Answer

The second overload delegates to the first:
suspend fun getRoute(fromLat: Double, fromLon: Double, toLat: Double, toLon: Double, profile: String = "bike") = getRoute(listOf(fromLat to fromLon, toLat to toLon), profile)
They can't be merged into one function with defaults because the parameter types differ (List<Pair> vs four Doubles) — these are true overloads, not default-value variants.

Chapter 3

String Templates

Kotlin has built-in string interpolation. You embed values directly in string literals using $variable or ${expression} — no String.format() needed.

Variable interpolation

String bbox = south + "," + west + "," + north + "," + east;
Log.w("MapViewModel", "POI load failed: " + it.getMessage());
// OverpassRepository.kt
val bbox = "$south,$west,$north,$east"

// MapViewModel.kt
Log.w("MapViewModel", "POI load failed: ${it.message}")

Expression interpolation with ${}

Wrap any expression in ${} to embed it inline.

Kotlin 
// Format remaining distance with one decimal
val label = "Remaining: ${"%.1f".format(remainingMeters / 1000.0)} km"

// Current instruction number out of total
val status = "Step ${currentIndex + 1} of ${instructions.size}"

// Nested property access
val msg = "Nearest junction: ${nearestNode.ref} at ${nearestNode.lat}, ${nearestNode.lon}"

Raw strings (triple quotes)

Triple-quoted strings span multiple lines and don't need escaping. Perfect for Overpass QL queries.

// Ugly escaping required
String query = "[out:json];relation[route=bicycle][network~\"ncn|rcn|lcn\"](" + bbox + ");out geom;";
// OverpassRepository.kt — no escaping, template still works
val query = """[out:json];relation[route=bicycle][network~"ncn|rcn|lcn"]($bbox);out geom;"""
🚲
In FietsNavNu: OverpassRepository.kt uses raw strings for two Overpass QL queries — one that fetches cycling junction nodes (rcn_ref) and one that fetches named cycling routes. The quotes inside the query don't need backslash-escaping.
💡 Exercise

Write a string template that produces a navigation instruction like:
"In 250 m: Turn left onto Overtoom"
given val distanceM = 250 and val text = "Turn left onto Overtoom".

Answer
Kotlin
val instruction = "In $distanceM m: $text"

Chapter 4

Data Classes

A data class is Kotlin's answer to the Java POJO. One line per field replaces 30+ lines of constructor, getters, equals(), hashCode(), and toString().

Java POJO → Kotlin data class

The Java Poi in pois-service is a textbook POJO:

// pois-service/src/main/java/com/fietsrouten/pois/model/Poi.java
public class Poi {
    private long id;
    private String name;
    private String amenity;
    private double lat;
    private double lon;

    public Poi(long id, String name, String amenity, double lat, double lon) {
        this.id = id; this.name = name; this.amenity = amenity;
        this.lat = lat; this.lon = lon;
    }

    public long getId()        { return id; }
    public String getName()    { return name; }
    public String getAmenity() { return amenity; }
    public double getLat()     { return lat; }
    public double getLon()     { return lon; }
    // no equals/hashCode/toString — you'd add those too
}
// app/src/main/kotlin/com/fietsrouten/data/model/Models.kt
data class Poi(
    val id: Long,
    val name: String,
    val amenity: String,
    val lat: Double,
    val lon: Double
)
// Compiler generates: constructor, getters, equals, hashCode, toString, copy

Default values in data classes

Fields can have defaults — callers that don't care about a field omit it.

Kotlin — Models.kt 
data class RouteResult(
    val coordinates: List<List<Double>>,
    val distanceMeters: Double,
    val durationMs: Long,
    val instructions: List<RouteInstruction>,
    val elevationProfile: List<Double> = emptyList()  // optional
)

// Callers that don't need elevation:
val result = RouteResult(coords, 12500.0, 3_600_000L, instructions)

Serialization annotations

Kotlin — Models.kt 
// JSON field "place_id" maps to Kotlin property placeId
data class NominatimResult(
    @SerializedName("place_id")    val placeId: Long,
    @SerializedName("display_name") val displayName: String,
    val lat: String,
    val lon: String

Destructuring

Data classes generate component1(), component2()… functions, enabling destructuring declarations.

Kotlin — NavigationEngine.kt 
// SegmentResult is a private data class with three fields
private data class SegmentResult(
    val index: Int,

// Destructure the returned SegmentResult in one line:

// Pairs also destructure — used in RouteRepository.kt:

copy() — non-destructive updates

Because data class fields are val, you never mutate them — you create a modified copy instead.

Kotlin 
val original = Poi(1L, "Café Central", "cafe", 52.37, 4.89)
val renamed  = original.copy(name = "Café Noord")   // only name changes
// original is unchanged
🚲
In FietsNavNu: Models.kt defines 12 data classes — from NominatimResult to CyclingRoute — all without a single getter method or constructor body. The file is 95 lines where the Java equivalent would be ~500.

Chapter 5

Null Safety

Kotlin's type system distinguishes nullable from non-nullable references at compile time. Most NullPointerExceptions become compile errors instead of runtime crashes.

Nullable types

A type is non-nullable by default. Appending ? allows null.

Kotlin 
val name: String  = "Café"   // can never be null
val name: String? = null     // ok

// From NavigationSession.kt:
val nextInstruction: RouteInstruction?  // null when on the last instruction

// From MapViewModel.kt:
private val _userLocation = MutableStateFlow<Pair<Double, Double>?>(null)
The compiler prevents calling methods on a nullable without a guard — nextInstruction.text is a compile error if the type is RouteInstruction?.

Safe call ?.

Returns null instead of throwing if the receiver is null.

// Explicit null guard every time
String ref = (el.getTags() != null) ? el.getTags().get("rcn_ref") : null;
// OverpassRepository.kt
val ref = el.tags?.get("rcn_ref")   // null if tags is null, otherwise the value

Elvis operator ?:

Provides a fallback value when the left side is null.

Kotlin — OverpassRepository.kt 
// Try "name" tag, fall back to "ref", bail out if both are null
val name = el.tags?.get("name") ?: el.tags?.get("ref") ?: return@mapNotNull null

// Provide a default string
val network = el.tags?.get("network") ?: "lcn"

let for null-conditional blocks

Execute a block only when the value is non-null, and use the value safely inside.

Kotlin — MapViewModel.kt 
// Returns null if _userLocation.value is null; otherwise builds a NominatimResult
fun currentLocationResult(): NominatimResult? =
    _userLocation.value?.let { (lat, lon) ->
        NominatimResult(
            placeId     = -1L,
            displayName = "Mijn locatie",
            lat         = lat.toString(),
            lon         = lon.toString()
        )
    }

takeIf — inline guard

Returns the value if the predicate holds, otherwise null. Chains naturally with ?..

Kotlin — OverpassRepository.kt 
// Drop geometries with fewer than 2 points, then map the rest
member.geometry
    ?.takeIf { it.size >= 2 }
    ?.map { pt -> listOf(pt.lon, pt.lat) }

lateinit var

For non-nullable properties that can't be initialised in the constructor — common with Android view binding.

Kotlin — MainActivity.kt 
class MainActivity : AppCompatActivity() {
    private lateinit var binding: ActivityMainBinding  // not null, but set later

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        binding = ActivityMainBinding.inflate(layoutInflater)  // initialised here
        setContentView(binding.root)
    }
}
🚲
In FietsNavNu: MapViewModel.kt exposes nearly all its state as nullable types (RouteResult?, NavigationSession?, etc.) because they start empty and are populated asynchronously. The compiler forces every call site to handle the null case — no more crashes when the route hasn't loaded yet.

Answer
Kotlin
// Exactly how MapViewModel.kt does it:

Chapter 6

Higher-Order Functions & Lambdas

Functions are first-class citizens in Kotlin. You can pass them as arguments, store them in variables, and return them. The collection API is built entirely on this — and the app uses it everywhere.

Lambda syntax

Kotlin 
// Full form
val double: (Int) -> Int = { x: Int -> x * 2 }

// Type inferred from context
val double = { x: Int -> x * 2 }

// Single parameter — use implicit 'it'
pois.filter { it.amenity == "cafe" }

// Trailing lambda: when the last argument is a lambda, move it outside ()
nodes.filter { node -> node.lat > 52.0 }
viewModelScope.launch { /* coroutine body */ }

Collection operations from the app

FunctionWhat it does
map { }Transform every element, return new list
filter { }Keep elements where predicate is true
any { }True if at least one element matches
mapNotNull { }Map + drop nulls in one step
filterIndexed { i, v }Filter with access to index
zipWithNext { a, b }Pair each element with the next
minOf / maxOf { }Min/max by a property
toMap()Build a Map from a list of Pairs
Kotlin — PoiServiceRepository.kt 
// map — transform GeoJSON features into Poi objects
api.getPois(south, west, north, east).features.map { f ->
    Poi(
        id      = f.properties.id,
        name    = f.properties.name,
        amenity = f.properties.amenity,
        lat     = f.geometry.coordinates[1],
        lon     = f.geometry.coordinates[0]
    )
}

// filter + any — keep only POIs within 150 m of the route
val sample = routeCoords.filterIndexed { i, _ -> i % 5 == 0 }
return pois.filter { poi ->
    sample.any { pt -> haversineMeters(poi.lat, poi.lon, pt[1], pt[0]) <= 150.0 }
}
Kotlin — MapViewModel.kt 
// zipWithNext — distance of each leg between selected knooppunten
fun getLegDistances(nodes: List<Knooppunt>): List<Double> =
    nodes.zipWithNext { a, b -> haversineMeters(a.lat, a.lon, b.lat, b.lon) }

// minOf / maxOf — bounding box around a route
val south = coords.minOf { it[1] }
val north = coords.maxOf { it[1] }
val west  = coords.minOf { it[0] }
val east  = coords.maxOf { it[0] }
Kotlin — OverpassRepository.kt 
// mapNotNull — map and skip null results in one pass
api.query(query).elements.mapNotNull { el ->
    if (el.type == "node" && el.lat != null && el.lon != null) {
        val ref = el.tags?.get("rcn_ref") ?: return@mapNotNull null
        Knooppunt(el.id, el.lat, el.lon, ref)
    } else null
}

Labelled return inside a lambda

Inside a lambda, bare return is not allowed — it would return from the enclosing function. Use return@label to exit only the lambda (continue to the next iteration).

Kotlin — OverpassRepository.kt 
elements.mapNotNull { el ->
    if (el.type != "relation") return@mapNotNull null  // skip this element
    val name = el.tags?.get("name") ?: return@mapNotNull null
    // ...
    CyclingRoute(el.id, name, network, segments)
}

Object-Oriented Kotlin

Kotlin classes are public and final by default. There is no static keyword — singletons use object, and static members use companion object.

Singleton objects

A top-level object declaration creates a thread-safe singleton, instantiated lazily on first use.

public class Config {
    public static final String GRAPHHOPPER_BASE_URL = "https://fietsnav-routing.learndelingo.nl";
    public static final String NOMINATIM_BASE_URL   = "https://fietsnav-geocoding.learndelingo.nl";
    // ...
    private Config() {}  // prevent instantiation
}
// Config.kt — accessed as Config.GRAPHHOPPER_BASE_URL, no instantiation
object Config {
    const val GRAPHHOPPER_BASE_URL = "https://fietsnav-routing.learndelingo.nl"
    const val NOMINATIM_BASE_URL   = "https://fietsnav-geocoding.learndelingo.nl"
    const val OVERPASS_BASE_URL    = "https://overpass-api.de/"
    const val POI_BASE_URL         = "https://fietsnav-pois.learndelingo.nl"
    const val MAP_STYLE_URL        = "https://tiles.openfreemap.org/styles/bright"
    const val MAP_STYLE_DARK_URL   = "https://basemaps.cartocdn.com/gl/dark-matter-gl-style/style.json"
}

companion object — static members

Static members live inside a companion object within the class. There is one companion per class.

Kotlin — NavigationEngine.kt

    private var lastBearing = 0f

Enums with properties

Kotlin enums are full classes — each constant can carry data.

when — smarter switch

when replaces switch and long if/else if chains. Crucially, it is an expression — it returns a value.

Kotlin 
// GraphHopper sign codes → drawable resource
val icon = when (instruction.sign) {
     0   -> R.drawable.ic_straight
    -2   -> R.drawable.ic_turn_left
     2   -> R.drawable.ic_turn_right
    -3   -> R.drawable.ic_sharp_left
     3   -> R.drawable.ic_sharp_right
    else -> R.drawable.ic_straight
}

// when with type checks
val message = when (error) {
    is IOException         -> "Network error"
    is IllegalStateException -> "App error: ${error.message}"
    else                   -> "Unknown error"
}

// when as a statement (no value needed)
when (plannerMode) {
    PlannerMode.ADDRESS     -> showAddressPanel()
    PlannerMode.KNOOPPUNTEN -> showKnooppuntenPanel()
}

runCatching — functional error handling

Wraps a block in a try/catch and returns a Result<T>. Chain .onSuccess, .onFailure, or .getOrDefault to handle each case.

    poisVisible.setValue(false);
}

runCatching {

    _pois.value = poiRepository.filterNearRoute(all, coords)
}.onFailure {
    Log.w("MapViewModel", "POI load failed: ${it.message}")
    _poisVisible.value = false
}

// getOrDefault — inline fallback value

Chapter 9

Coroutines

Coroutines are Kotlin's answer to async programming. A suspending function can pause without blocking a thread, then resume when its result is ready — so async code looks and reads like synchronous code.

The suspend keyword

Marking a function suspend lets it pause mid-execution. From the caller's perspective it still looks like a normal blocking call — no callbacks, no .then().

// Java Retrofit — callback hell
api.search(query).enqueue(new Callback<List<NominatimResult>>() {
    @Override public void onResponse(Call<...> call, Response<...> response) {
        List<NominatimResult> results = response.body();
        // do something
    }
    @Override public void onFailure(Call<...> call, Throwable t) { }
});
// Kotlin Retrofit — reads like synchronous code
// RouteRepository.kt
suspend fun searchAddress(query: String): List<NominatimResult> =
    nominatimApi.search(query)   // pauses here, resumes with the list

Coroutine builders: launch

suspend functions can only be called from another suspend function or a coroutine builder. launch starts a fire-and-forget coroutine scoped to a lifecycle.

Kotlin — MapViewModel.kt 
fun calculateRoute() {
    val from = fromLocation ?: run { _error.value = "Selecteer eerst een startadres."; return }
    val to   = toLocation   ?: run { _error.value = "Selecteer eerst een bestemming."; return }

    viewModelScope.launch {          // starts a coroutine in the ViewModel's scope
        _isLoading.value = true
        _error.value = null

        val results = fetchAllProfiles { profile ->   // suspend call — ok inside launch
            repository.getRoute(
                fromLat = from.lat.toDouble(), fromLon = from.lon.toDouble(),
                toLat   = to.lat.toDouble(),   toLon   = to.lon.toDouble(),
                profile = profile
            )
        }

        if (!applyRouteResults(results)) _error.value = "Route niet gevonden"
        _isLoading.value = false
    }
    // launch returns immediately; the body runs asynchronously
}
viewModelScope is cancelled automatically when the ViewModel is cleared — any in-flight network requests are cancelled too. No memory leaks.

async / await — parallel work

async starts a coroutine that returns a Deferred<T>. Call .await() to collect the result. Multiple async blocks run concurrently — total time equals the slowest, not the sum.

Kotlin — MapViewModel.kt 
// Fetch all three cycling profiles in parallel
private suspend fun fetchAllProfiles(
    request: suspend (profile: String) -> RouteResult
): Map<CyclingProfile, RouteResult> = coroutineScope {
    CyclingProfile.values()
        .map { profile ->
            // Each async block starts immediately — all three run concurrently
            profile to async { runCatching { request(profile.apiName) } }
        }
        .mapNotNull { (profile, deferred) ->
            deferred.await().getOrNull()?.let { profile to it }
        }
        .toMap()
}
// coroutineScope waits for all children before returning

Structured concurrency

Coroutines form a parent-child tree. When a parent is cancelled, all children are cancelled too. This prevents resource leaks automatically.

Scope tree in FietsNavNu 
viewModelScope                        // cancelled when ViewModel clears
  └── launch { calculateRoute }
        └── coroutineScope { fetchAllProfiles }
              ├── async { bike route request }
              ├── async { mtb route request }
              └── async { racingbike route request }
🚲
In FietsNavNu: without async, fetching all three cycling profiles would take ~3× longer (sequential). With async, all three GraphHopper API calls fly in parallel. If the user navigates away mid-request, viewModelScope cancels everything — no orphaned threads.
💡 Exercise

What would happen if calculateRoute() called repository.getRoute() without wrapping it in viewModelScope.launch { }?

Answer

A compile error. repository.getRoute() is a suspend function and can only be called from a coroutine or another suspend function. calculateRoute() is a regular function called from the UI, so it is not in a coroutine context. viewModelScope.launch { } creates the coroutine that makes the suspend call legal.

Chapter 10

Flow & StateFlow

Coroutines handle a single async value. Flow handles a stream of values over time. StateFlow is a hot Flow that always holds the latest value — reactive state for the UI.

StateFlow as UI state

Every piece of observable state in MapViewModel follows the same pattern:

Kotlin — MapViewModel.kt 
// Internal — writable
private val _isLoading = MutableStateFlow(false)
private val _route     = MutableStateFlow<RouteResult?>(null)
private val _pois      = MutableStateFlow<List<Poi>>(emptyList())

// Public — read-only view exposed to the Fragment
val isLoading: StateFlow<Boolean>      = _isLoading
val route:     StateFlow<RouteResult?> = _route
val pois:      StateFlow<List<Poi>>   = _pois

// Writing (ViewModel only):
_isLoading.value = true

// Reading (Fragment):
viewLifecycleOwner.lifecycleScope.launch {
    viewModel.isLoading.collect { loading ->
        progressBar.isVisible = loading
    }
}
The underscore prefix (_isLoading / isLoading) is a Kotlin convention. It makes the encapsulation visible at a glance.

Building a reactive search with operators

The address search in MapViewModel chains three Flow operators to debounce keystrokes, cancel stale requests, and emit results.

Kotlin — MapViewModel.kt 
private val fromQuery = MutableStateFlow("")

init {
    viewModelScope.launch {
        fromQuery
            .debounce(300)           // 1. wait 300 ms of silence before acting
            .flatMapLatest { query ->
                flow {               // 2. for each new query, start a fresh search
                    if (query.length >= 3)
                        emit(runCatching { repository.searchAddress(query) }
                                .getOrDefault(emptyList()))
                    else
                        emit(emptyList())
                }
            }                        // flatMapLatest cancels the previous inner flow
            .collect { _fromSuggestions.value = it }  // 3. push results to UI
    }
}

// Called by the Fragment on every keystroke:
fun searchFrom(query: String) { fromQuery.value = query }
OperatorWhat it does here
debounce(300)Drops emissions arriving within 300 ms — avoids a network call per keystroke
flatMapLatest { }Starts a new inner flow per query; cancels the previous one if a newer query arrives
collect { }Receives each final value and writes it to the backing StateFlow

flow { } builder

Creates a cold stream — it only starts running when collected, and each collector gets its own independent execution.

Kotlin 
// A cold flow that emits one search result
flow {
    emit(repository.searchAddress(query))  // suspends here during the network call
}

// vs StateFlow — hot, starts running immediately, replays latest to new collectors
val isLoading: StateFlow<Boolean> = _isLoading
🚲
In FietsNavNu: MapViewModel has 14 StateFlow properties — route, navigation session, loading state, POI list, cycling routes, planner mode, selected nodes, and more. Every UI update is driven by collecting these flows in the Fragment rather than imperatively calling setter methods. Adding a new UI element only requires one new collect call.
💡 Exercise

Why is _fromSuggestions declared as MutableStateFlow privately, while the public fromSuggestions is typed as StateFlow? What happens if the Fragment tries to cast it back to MutableStateFlow?

Answer

Encapsulation. The ViewModel owns the mutation; the Fragment should only observe. Typing the public property as the read-only StateFlow interface enforces this at the type level.

A Fragment can cast it: (viewModel.fromSuggestions as MutableStateFlow).value = ... — but that is an explicit, ugly workaround that signals a design violation. In production code, you'd add a @Suppress("UNCHECKED_CAST") lint suppression, making the intent clear.

Chapter 11

Java Interop

Kotlin runs on the JVM alongside Java. Mixing both in one project — as FietsNavNu does — is seamless. This chapter covers the edge cases you'll hit when converting the Java pois-service to Kotlin.

Calling Java from Kotlin

Kotlin calls Java classes directly. Getters and setters are exposed as properties.

Kotlin calling Java 
// Java Location class — Kotlin exposes getters as properties
val speed   = location.speed        // calls location.getSpeed()
val bearing = location.bearing      // calls location.getBearing()
val lat     = location.latitude     // calls location.getLatitude()

// Android Log — static Java methods called directly
Log.w("Tag", "message")

Platform types

When Java code has no nullability annotations, Kotlin exposes the return as a platform type (written String!). You are responsible for null checks — the compiler doesn't enforce them.

Kotlin — NavigationEngine.kt 
// location.hasBearing() is a Java method — guard before using .bearing
if (location.hasBearing() && location.speed > 0.5f) {
    lastBearing = location.bearing
}

@JvmStatic and @JvmField

When Java code needs to call a companion object member, annotate it so it appears as a true static from Java's perspective.

Kotlin 
class NavigationEngine {
    companion object {
        @JvmField  // Java: NavigationEngine.OFF_ROUTE_THRESHOLD_METERS
        val OFF_ROUTE_THRESHOLD_METERS = 30.0

        @JvmStatic  // Java: NavigationEngine.create()
        fun create(): NavigationEngine = NavigationEngine()
    }
}

// Without @JvmStatic, Java would need:
// NavigationEngine.Companion.create()

Converting pois-service to Kotlin

The full backend conversion — three files change entirely. Spring annotations work identically in Kotlin.

@RestController
@RequestMapping("/api/pois")
public class PoiController {
    private final PoiRepository repository;

    public PoiController(PoiRepository repository) {
        this.repository = repository;
    }

    @GetMapping
    public ResponseEntity<Map<String, Object>> getPois(
            @RequestParam double south, @RequestParam double west,
            @RequestParam double north, @RequestParam double east,
            @RequestParam(defaultValue = "cafe,restaurant,fast_food") String types) {
        List<String> typeList = Arrays.asList(types.split(","));
        List<Poi> pois = repository.findInBbox(south, west, north, east, typeList);
        List<Map<String, Object>> features = pois.stream()
                .map(poi -> Map.of(
                        "type", "Feature",
                        "geometry", Map.of("type", "Point",
                                "coordinates", new double[]{poi.getLon(), poi.getLat()}),
                        "properties", Map.of("id", poi.getId(),
                                "name", poi.getName(), "amenity", poi.getAmenity())))
                .collect(Collectors.toList());
        return ResponseEntity.ok(Map.of("type", "FeatureCollection", "features", features));
    }
}
@RestController
@RequestMapping("/api/pois")
class PoiController(private val repository: PoiRepository) {  // constructor injection in header

    @GetMapping
    fun getPois(

        @RequestParam west: Double,
        @RequestParam north: Double,
        @RequestParam east: Double,

💡 Exercise

If you convert Poi.java to a Kotlin data class, does the existing Java PoiRepository that calls poi.getId() and poi.getName() still compile without changes?

Answer

Yes. Kotlin data classes generate standard JVM getters (getId(), getName()…) for each val property. Java code that calls these getters sees no difference.

The only time you'd need to add @JvmField is if Java code tries to access the field directly without a getter — e.g. poi.id — which is uncommon in Spring/JPA patterns.
